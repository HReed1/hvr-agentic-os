# Blog Publication & Validation Pipeline

**Trigger Constraint:** Execute this pipeline whenever the user requests generation of a blog post, or explicitly invokes the "Master Co-Author" framework.

**Purpose:** To systematically draft highly human-centric architectural retrospectives, coupled with 100% technically authentic Git-verified simulation scenarios, entirely isolated from the Orchestration Swarm.

## Workflow Execution Steps

### Phase 1: Contextual Audit & The Interview
1. **The Anchor:** Read `master_index.json` internally using the `view_file` tool. Identify the explicit JSON tree for the requested Era/Pivot. Note the foundational `commit`, `philosophical_pivot`, and theoretical `simulation_scenario`.
2. **The Interview:** Prompt the user with exactly **3 highly-targeted questions**. 
    - Question 1: Extract the human frustration/anxiety mapping to the `philosophical_pivot`.
    - Question 2: Extract the architectural mechanism.
    - Question 3: Extract the specific "Aha!" realization.
3. **Patience Loop:** Cease generation immediately after prompting. Wait for the user's explicit reply.

### Phase 2: The drafting
1. Once the user replies, synthesize the human context with the parsed technical constraints.
2. Draft the blog post strictly utilizing the 4-part structure inside `docs/blog/conversations/`:
    - **1. The Hook:** Emotional/strategic reality.
    - **2. The Architecture:** The technical shift.
    - **3. The Breakthrough:** The specific mechanism that resolved the hurdle.
    - **4. The Reader Simulation:** An interactive HTML `<details>` element testing the reader. *Do not guess real code yet; just place the structural HTML and a placeholder "Naive Code" & "Solution" diff block.*

### Phase 3: The Technical Authenticity Audit
1. **Git Synchronization:** Natively execute `git show <commit_hash>` using the `run_command` tool strictly targeting the commit defined in `master_index.json`.
2. **Surgical Mutation:** Evaluate the physical `git diff` generated by the terminal. Map those precise lines of code directly into the "Naive Code" (the state before the commit) and "Solution" (the exact `diff` of the fix) blocks within the drafted markdown file. 
3. **Validation Assertion:** The IDE must inherently detect if the drafted `simulation_scenario` maps directly to the historical physical Git commit logic. If they diverge, gracefully execute an automated Surgical Mutation using the `replace_file_content` tool to anchor Section 4 strictly to reality.

### Phase 4: Completion State Check
1. Evaluate the drafted document for loose ends. If you detect placeholder text or incomplete assumptions, prompt the user explicitly by alerting them that "the posts need more user input".
