# TDAID Testing & Cryptographic Guardrails

When operating resolving Python testing frameworks within the Zero-Trust `.staging/` environment natively, Swarm Agents MUST adhere strictly to the following parameters:

1. **Test Runner Capability Delegation**
   The QA Engineer is the sole test author. It must write the `.staging/tests/` script natively and explicitly invoke the `execute_tdaid_test` tool to initiate validation matrices natively. The Executor is structurally restricted to writing Functional Logic and must evaluate the failing traceback generated by the QA Engineer.

2. **The QA Testing Validation Matrix**
   The QA Engineer physically holds the `execute_tdaid_test` tool. This is the **only** authorized mechanism for asserting local functional tests against `.staging`.
   - The test script path MUST be relative to the `.staging` airspace when resolving the execution.

3. **Cryptographic Signatures (`.qa_signature`)**
   Upon a Pytest run exiting with a standard Code 0 (Success), the `execute_tdaid_test` tool will natively write an HMAC sha256 structural proxy cache to `.staging/.qa_signature`.

4. **Red/Green Development Loops**
   Under the Spec-Driven TDAID paradigm, the QA Engineer MUST author the test first and execute it natively, expecting it to fail (The Red Baseline Phase). The QA Engineer MUST NEVER output `[QA PASSED]` on this initial generation. It MUST explicitly return the negative `[QA REJECTED]` traceback to transfer testing context bounds to the Executor. The Executor then attempts to turn the trace into a Green Exit 0 by drafting the functional `api/` or DOM logic natively.
